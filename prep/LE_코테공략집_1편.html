CH 01

🧱 코테 기초 개념 — 아무것도 몰라도 되는 출발점

"시간복잡도가 뭔지 모르겠다"면 이 장부터. 여기 나온 개념만 알면 뒤의 모든 설명이 그대로 읽힙니다. 이미 아는 부분은 건너뛰세요.

1. 시간복잡도(Time Complexity) — 딱 이것만

한 문장 정의: 입력(n)이 커질 때 계산 횟수가 얼마나 빨리 불어나는지를 나타내는 지표. "몇 초 걸리나"가 아니라 증가 속도를 봅니다.

왜 필요한가: 코딩테스트엔 시간 제한(보통 1~10초)이 있습니다. 컴퓨터는 대략 1초에 1억(10⁸)번의 단순 연산을 합니다. 내 풀이의 연산 횟수가 이걸 크게 넘으면 로직이 맞아도 "시간 초과"로 오답 처리됩니다. 그래서 코딩을 시작하기 전에 "이 방법이면 대략 몇 번 계산하지?"를 어림하는 습관이 필요합니다.

Big-O 표기법: 연산 횟수에서 가장 빨리 커지는 항만 남기고 나머지는 버리는 표기법입니다. 예: 연산 횟수가 3n² + 5n + 2번이라면, n이 커질수록 n² 항이 지배하므로 O(n²)라고 씁니다.

표기이름대표 예n=10만일 때 연산 수
O(1)상수배열 인덱스 접근, 해시맵 조회1
O(log n)로그이분탐색약 17
O(n)선형배열 한 번 훑기10만
O(n log n)선형로그정렬(Arrays.sort)약 170만
O(n²)제곱이중 for문100억 → ⚠️ 시간 초과
O(2ⁿ)지수부분집합 전부 나열n=20만 돼도 약 100만
O(n!)팩토리얼순열 전부 나열n=10만 돼도 약 360만

💡 읽는 법: O(n²)은 "입력이 10배면 계산은 100배". 반대로 O(log n)은 반씩 쪼개므로 입력이 100만이어도 약 20번이면 끝.

실전 사용법 — 제약조건 보고 알고리즘 고르기

문제에 적힌 입력 크기(제약조건)를 보면 어떤 복잡도까지 허용되는지 역산할 수 있습니다. 이게 유형 선택의 출발점입니다.

n 크기허용 복잡도후보 알고리즘

2. 공간복잡도 — 30초면 충분

같은 개념의 메모리 버전입니다. 예: 길이 n짜리 배열 = O(n), n×n 2차원 배열 = O(n²). 코테에선 보통 시간이 먼저 터지므로 "2차원 배열 크기가 억 단위면 위험" 정도의 감각이면 됩니다.

4. 자주 나오는 용어 (뒤에서 계속 등장)

  • 정점(node) · 간선(edge): 그래프의 점과 선. 미로의 칸 = 정점, 이웃 칸끼리의 연결 = 간선.
  • 방문 배열(visited): 이미 가 본 곳 표시. 없으면 같은 곳을 무한 반복.
  • 점화식: "현재 답 = 이전 답들의 조합"인 식. 예: dp[i] = dp[i-1] + dp[i-2]. DP의 심장.
  • 메모이제이션: 한 번 계산한 답을 표에 저장해서 재계산을 막는 것.
  • 엣지 케이스 / 반례: 경계 상황(빈 입력, n=1, 최댓값). LE는 점수를 안 보여주므로 스스로 만들어 검증해야 함.
  • 오버플로: int는 최대 약 21억. 합계·곱셈이 커지면 long.

3. BFS

📖 개념: BFS(너비 우선 탐색)는 시작점에서 가까운 칸부터 물결처럼 한 겹씩 퍼져나가는 탐색입니다. 거리 1인 칸을 전부 본 뒤에야 거리 2를 보기 때문에, 어떤 칸에 처음 도착한 순간이 곧 최단 거리임이 보장됩니다(모든 이동 비용이 같을 때). "먼저 온 순서대로" 처리해야 하므로 큐(줄서기)를 씁니다.

🔎 언제: "최소 이동 횟수/최단 거리"인데 모든 이동 비용이 동일(=1). 격자·미로·전파(불/바이러스)·레벨 단위 확산.

🧭 접근: 시작점을 큐에 넣고 거리 0 → 큐에서 꺼내 4방향 확장, 처음 도달할 때가 최단.

⚠️ 함정: 방문 표시는 큐에 넣을 때 해야 중복 방지. 거리 배열을 방문표시로 겸하면 깔끔.

int bfs(int[][] maps) {
    int R = maps.length, C = maps[0].length;
    int[][] dist = new int[R][C];
    for (int[] row : dist) Arrays.fill(row, -1);
    ArrayDeque<int[]> q = new ArrayDeque<>();
    q.offer(new int[]{0,0}); dist[0][0] = 0;
    int[] dr={-1,1,0,0}, dc={0,0,-1,1};
    while (!q.isEmpty()) {
        int[] c = q.poll();
        for (int d=0; d<4; d++) {
            int nr=c[0]+dr[d], nc=c[1]+dc[d];
            if (nr<0||nr>=R||nc<0||nc>=C) continue;
            if (maps[nr][nc]==0 || dist[nr][nc]!=-1) continue;
            dist[nr][nc] = dist[c[0]][c[1]] + 1;
            q.offer(new int[]{nr,nc});
        }
    }
    return dist[R-1][C-1];
}

시간복잡도: O(V+E). 격자면 O(R·C).

4. DFS / flood fill

📖 개념: DFS(깊이 우선 탐색)는 한 방향으로 끝까지 파고들다 막히면 되돌아와 다른 길을 가는 탐색입니다. 미로에서 한쪽 벽에 손을 짚고 계속 걷는 것과 같습니다. "서로 연결된 것 몽땅 찾기"(섬·덩어리 세기)에 자연스럽고, 보통 재귀(자기 자신을 호출하는 함수)로 짧게 구현합니다. 시간복잡도는 BFS와 같은 O(V+E).

🔎 언제: "연결된 덩어리 개수", 영역 크기, 섬 개수, 경로 존재 여부. 깊이 우선으로 끝까지 파고듦.

🧭 접근: 방문 안 한 칸에서 DFS 시작 → 재귀로 사방 전파 → 한 번의 DFS = 한 덩어리.

⚠️ 함정: 격자가 크면 재귀 깊이 스택오버플로 → 스택 자료구조로 전환 고려. 방문 체크 필수.

int R, C; int[][] g; boolean[][] vis;
void dfs(int r, int c) {
    vis[r][c] = true;
    int[] dr={-1,1,0,0}, dc={0,0,-1,1};
    for (int d=0; d<4; d++) {
        int nr=r+dr[d], nc=c+dc[d];
        if (nr<0||nr>=R||nc<0||nc>=C) continue;
        if (g[nr][nc]==0 || vis[nr][nc]) continue;
        dfs(nr, nc);
    }
}
// 덩어리 수 세기
int count = 0;
for (int i=0;i<R;i++) for (int j=0;j<C;j++)
    if (g[i][j]==1 && !vis[i][j]) { dfs(i,j); count++; }

7. 다익스트라

📖 개념: 다익스트라는 BFS의 업그레이드입니다. BFS는 모든 이동 비용이 같을 때만 최단을 보장하는데, 길마다 비용이 다르면 "지금까지 가장 싸게 도달한 곳"부터 차례로 확정해 나가야 합니다. 그 "가장 싼 곳 먼저"를 위해 우선순위 큐를 씁니다.

🔎 언제: "최단 거리/최소 비용"인데 간선마다 비용이 다름(양수). BFS로 안 되는 가중 그래프.

🧭 접근: dist[]를 INF로 초기화 → 시작점 0 → PQ에서 최소 비용 노드 꺼내 완화(relax).

⚠️ 함정: 음수 간선이면 다익스트라 불가(벨만포드). dist는 long 고려. 꺼낸 게 이미 갱신된 값보다 크면 skip.

int[] dijkstra(List<int[]>[] adj, int start, int n) {
    int[] dist = new int[n];
    Arrays.fill(dist, Integer.MAX_VALUE);
    dist[start] = 0;
    PriorityQueue<int[]> pq = new PriorityQueue<>((a,b)->a[1]-b[1]);
    pq.offer(new int[]{start, 0});
    while (!pq.isEmpty()) {
        int[] cur = pq.poll();
        int u = cur[0], d = cur[1];
        if (d > dist[u]) continue;          // 오래된 값 skip
        for (int[] e : adj[u]) {            // e = {v, w}
            int v = e[0], w = e[1];
            if (dist[u] + w < dist[v]) {
                dist[v] = dist[u] + w;
                pq.offer(new int[]{v, dist[v]});
            }
        }
    }
    return dist;
}

시간복잡도: O(E log V).

8. DP

📖 개념: DP(동적계획법)는 큰 문제의 답을 작은 문제의 답으로 조립하는 기법입니다. 피보나치를 그냥 재귀로 짜면 같은 계산을 수없이 반복하지만(O(2ⁿ)), 한 번 구한 답을 표(dp 배열)에 저장해두면 각 항을 한 번씩만 계산합니다(O(n)). 그 조립 공식이 점화식(예: dp[i] = dp[i-1] + dp[i-2])입니다.

🔎 언제: "이전 답으로 현재 답", 겹치는 부분문제 + 최적 부분구조. 계단·피보나치·최장부분수열·경로 수·최소 비용.

🧭 접근: (1) 상태 정의 dp[i]=… (2) 점화식 (3) 초기값 (4) 계산 순서. "dp[i]가 뭘 의미하는지" 한 문장으로 말할 수 있어야 함.

⚠️ 함정: 상태 정의 모호하면 다 틀림. 배열 크기/초기값. 합은 long.

// 예: 1D DP — 계단 오르기 (1칸/2칸)
long[] dp = new long[n+1];
dp[0] = 1; dp[1] = 1;
for (int i=2; i<=n; i++) dp[i] = dp[i-1] + dp[i-2];

// 예: 2D DP — 격자 경로 최소합
int[][] dp = new int[R][C];
dp[0][0] = g[0][0];
for (int i=0;i<R;i++) for (int j=0;j<C;j++) {
    if (i==0 && j==0) continue;
    int best = Integer.MAX_VALUE;
    if (i>0) best = Math.min(best, dp[i-1][j]);
    if (j>0) best = Math.min(best, dp[i][j-1]);
    dp[i][j] = best + g[i][j];
}

9. 배낭 DP

📖 개념: DP의 대표 응용문제입니다. "용량이 제한된 배낭에 가치가 최대가 되도록 물건 담기". 물건마다 넣을까/말까를 전부 시도하면 O(2ⁿ)이지만, dp[용량]="이 용량으로 만들 수 있는 최대 가치" 표를 채우면 O(물건 수 × 용량)으로 끝납니다.

🔎 언제: "무게/용량 한정, 가치 최대". 0/1 배낭(각 물건 1개), 부분합 만들기.

🧭 접근: dp[w] = 용량 w일 때 최대 가치. 물건 루프 안에서 용량은 역순(0/1 배낭 핵심).

⚠️ 함정: 용량 정순으로 돌면 같은 물건 중복 사용됨(그건 무한 배낭). 0/1은 반드시 역순.

int[] dp = new int[cap+1];
for (int[] item : items) {        // item = {weight, value}
    int wt = item[0], val = item[1];
    for (int w = cap; w >= wt; w--)       // 역순!
        dp[w] = Math.max(dp[w], dp[w-wt] + val);
}
return dp[cap];

10. 백트래킹

📖 개념: 모든 경우의 수를 나무(트리) 모양으로 하나씩 시도하되, 조건상 틀려버린 가지는 즉시 포기하고 되돌아가는(backtrack) 기법입니다. "고르고 → 더 깊이 들어가고 → 돌아와서 고른 걸 취소" 3박자가 핵심. 경우의 수가 지수적으로 늘어나므로 N이 작을 때(≤20)만 씁니다.

🔎 언제: "모든 경우의 수", 순열·조합·부분집합, N-Queen, 조건 만족 조합 찾기. N이 작음(≤ 약 20).

🧭 접근: 선택 → 재귀 → 되돌리기(undo). 가지치기(pruning)로 불가능한 가지 조기 차단.

⚠️ 함정: 되돌리기 누락 → 오답. 방문/사용 배열 상태 복원 필수. 지수 시간이라 N 확인.

// 조합: n개 중 r개 고르기
void comb(int start, int n, int r, List<Integer> pick, List<List<Integer>> res) {
    if (pick.size() == r) { res.add(new ArrayList<>(pick)); return; }
    for (int i = start; i < n; i++) {
        pick.add(i);
        comb(i + 1, n, r, pick, res);
        pick.remove(pick.size() - 1);   // 되돌리기
    }
}

// 순열: used[] 사용
void perm(int[] arr, boolean[] used, List<Integer> cur, List<List<Integer>> res) {
    if (cur.size() == arr.length) { res.add(new ArrayList<>(cur)); return; }
    for (int i = 0; i < arr.length; i++) {
        if (used[i]) continue;
        used[i] = true; cur.add(arr[i]);
        perm(arr, used, cur, res);
        used[i] = false; cur.remove(cur.size()-1);   // 되돌리기
    }
}

11. 누적합

📖 개념: 구간 합을 매번 더하면 조회마다 O(n)이지만, "처음부터 i까지의 합"을 미리 배열로 만들어 두면 어떤 구간 합도 뺄셈 한 번(O(1))으로 나옵니다. "구간 [l,r]의 합을 여러 번 묻는" 문제의 정답.

🔎 언제: "구간 [l,r] 합을 여러 번 조회", 2D 부분합, 이동평균. 미리 계산해두고 O(1) 조회.

🧭 접근: prefix[i] = prefix[i-1] + a[i]. 구간합 = prefix[r] - prefix[l-1].

⚠️ 함정: 인덱스 off-by-one. 합은 long.

long[] pre = new long[n+1];
for (int i=1; i<=n; i++) pre[i] = pre[i-1] + a[i-1];
long sum = pre[r] - pre[l-1];   // 1-indexed 구간 [l,r]

// 2D 누적합
long[][] p = new long[R+1][C+1];
for (int i=1;i<=R;i++) for (int j=1;j<=C;j++)
    p[i][j] = g[i-1][j-1] + p[i-1][j] + p[i][j-1] - p[i-1][j-1];

12. 이분탐색 / 파라메트릭 서치

📖 개념: 업다운 숫자 맞히기 게임입니다. 정렬된 범위에서 가운데를 확인하고 절반을 통째로 버리기를 반복 — 100만 개도 약 20번(log₂ 100만)이면 찾습니다. 응용인 파라메트릭 서치는 "정답이 될 수 있는 값" 자체를 이분탐색하는 것.

🔎 언제: 정렬된 배열에서 값 찾기. 또는 "~을 최대/최소로 하는 최적값"(파라메트릭): 랜선 자르기, 예산, 최소 최대거리.

🧭 접근: 정답을 이분. check(mid)가 만족하면 범위를 좁힘. "가능/불가능"이 단조로울 때만.

⚠️ 함정: mid = lo + (hi-lo)/2 (오버플로 방지). lo/hi 갱신 방향, 종료 조건.

// lower_bound: target 이상이 처음 나오는 위치
int lowerBound(int[] a, int target) {
    int lo = 0, hi = a.length;
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (a[mid] < target) lo = mid + 1;
        else hi = mid;
    }
    return lo;
}

// 파라메트릭: "조건 만족하는 최댓값" 찾기
int lo = 1, hi = MAX, ans = 0;
while (lo <= hi) {
    int mid = lo + (hi - lo) / 2;
    if (possible(mid)) { ans = mid; lo = mid + 1; }
    else hi = mid - 1;
}

14. Union-Find (보너스)

📖 개념: "두 원소가 같은 그룹인가?"를 빠르게 답하는 자료구조입니다. 각 그룹의 대표(뿌리)를 기억해두고, find로 대표를 찾고 union으로 두 그룹을 합칩니다.

🔎 언제: "같은 그룹인가", 사이클 판별, 연결 요소, 크루스칼 MST. 도형/그래프 문제에서 종종 유용.

int[] parent;
int find(int x) { return parent[x]==x ? x : (parent[x]=find(parent[x])); }
void union(int a, int b) { parent[find(a)] = find(b); }
// 초기화: for(i) parent[i]=i;

        int[] dr = {0, 1, 0, -1};   // 우,하,좌,상
        int[] dc = {1, 0, -1, 0};
        int r = 0, c = 0, dir = 0;
        for (int v = 1; v <= n * n; v++) {
            a[r][c] = v;                        // 현재 칸 채우기
            int nr = r + dr[dir], nc = c + dc[dir];
            if (nr < 0 || nr >= n || nc < 0 || nc >= n || a[nr][nc] != 0) {
                dir = (dir + 1) % 4;            // 벽 or 이미 채움 => 시계방향 꺾기
                nr = r + dr[dir]; nc = c + dc[dir];
            }
            r = nr; c = nc;                     // 이동
        }
        return a;
    }
}

해설 (읽기용) - 값 v를 1→n²까지 하나씩 놓으면서 현재 방향으로 전진. - 전진할 칸이 격자 밖이거나 이미 채워진 칸(≠0)이면 방향을 시계방향으로 한 번 꺾음. - a[nr][nc] != 0 체크가 핵심 — 이게 나선의 "안쪽으로 감기"를 만든다. - 방향 순서를 {우,하,좌,상}으로 두면 시계방향. 반시계면 순서만 바꾸면 됨.

⚠️ 함정 - 마지막 값(n²)을 놓은 뒤에도 이동을 시도하는데, 루프가 끝나므로 문제없음. 단 v <= n*n 경계 확인. - 0을 "빈 칸" 표식으로 쓰므로 채우는 값은 1부터. 0부터 채우면 이 방식이 깨짐(빈칸 구분 불가) → visited 배열 별도 사용.

문제 2. 파스칼 삼각형

정수 n이 주어질 때, 파스칼 삼각형 n행을 반환하세요(i행은 i+1개 원소). 각 원소는 바로 위 두 수의 합, 양 끝은 1. 예: n=5 → [[1],[1,1],[1,2,1],[1,3,3,1],[1,4,6,4,1]]

정답 (검증됨)

class Solution {
    public int[][] solution(int n) {
        int[][] t = new int[n][];
        for (int i = 0; i < n; i++) {
            t[i] = new int[i + 1];              // i행은 i+1칸 (가변 길이)
            t[i][0] = t[i][i] = 1;              // 양 끝은 1
            for (int j = 1; j < i; j++)
                t[i][j] = t[i - 1][j - 1] + t[i - 1][j];  // 위 두 수 합
        }
        return t;
    }
}

해설 (읽기용) - 가변 길이 2차원 배열(new int[n][] 후 행마다 크기 지정)이 포인트 — 직사각형 배열이 아님. - 점화식 t[i][j] = t[i-1][j-1] + t[i-1][j]는 사실 DP. 도형이지만 DP 사고가 들어감. - 양 끝(j=0, j=i)은 1로 먼저 세팅하고, 내부만 합으로 채움.

⚠️ 함정 - t[i-1][...] 접근 시 i=0행에서 내부 루프(j<i)가 안 도므로 인덱스 에러 없음. 경계 자연 처리. - 값이 커지면(파스칼은 조합수라 폭발) long 필요. n이 크면 오버플로 주의.

문제 3. 지그재그(뱀) 행렬

정수 n, m이 주어질 때, n × m 행렬을 1부터 뱀처럼 지그재그로 채워 반환하세요. 짝수 행(0-index)은 왼→오, 홀수 행은 오→왼. 예: n=3, m=4 1 2 3 4 8 7 6 5 9 10 11 12

정답 (검증됨)

class Solution {
    public int[][] solution(int n, int m) {
        int[][] a = new int[n][m];
        int v = 1;
        for (int i = 0; i < n; i++) {
            if (i % 2 == 0)                     // 짝수 행: 왼 -> 오
                for (int j = 0; j < m; j++) a[i][j] = v++;
            else                                // 홀수 행: 오 -> 왼
                for (int j = m - 1; j >= 0; j--) a[i][j] = v++;
        }
        return a;
    }
}

해설 (읽기용) - 행 번호의 홀짝으로 방향만 뒤집으면 지그재그 완성. 방향배열까지 갈 필요 없음. - v++는 "현재 v를 넣고 v를 1 증가" — 채우기 카운터의 국룰.

⚠️ 함정 - "열 방향" 지그재그(세로로 뱀)면 i/j 역할을 바꾸면 됨. 지문에서 방향을 꼭 확인. - 대각선 지그재그(예: 지그재그 변환) 유형은 규칙이 다르니 별개.

🎯 도형 유형 시험장 체크

  • 방향배열 {우,하,좌,상} 손코딩
  • "벽 or 채워짐 → dir=(dir+1)%4" 패턴
  • 채우기 카운터 v++, 빈칸 표식(0) 충돌 주의
  • 가변 길이 배열(삼각형)은 new int[n][]
  • 제출 전 작은 예제(n=3,4)를 손으로 그려 대조 (LE는 점수 비공개!)

추가 연습: 백준 2941/2566(행렬 최댓값 위치), 2563/2477(도형 면적), 프로그래머스 "행렬의 곱셈"·"직사각형" 계열.

🗄️ SQL 치트시트 (⭐ Oracle)

LE 코테 SQL은 Oracle. MySQL 습관(LIMIT/IFNULL/DATE_FORMAT)이 그대로 나가면 에러. 지하철에서 이 한 장만 반복해서 눈에 바르면 Lv.2~3 SQL은 대부분 커버.

⏱️ 시험장 SQL 30분 컷 루틴

  1. 요구사항에서 출력 컬럼 · 정렬 · 필터를 먼저 밑줄 (지문 독해가 절반)
  2. FROM/JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY 순으로 뼈대
  3. 그룹별 1위/순위 요구면 윈도우함수 먼저 떠올리기
  4. 제출 전: 정렬 기준·동점 처리·NULL 포함 여부 재확인

1. 절 실행 순서 (⭐ 가장 중요)

작성 순서와 실행 순서가 다르다.

작성:  SELECT → FROM → JOIN → WHERE → GROUP BY → HAVING → ORDER BY
실행:  FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY

결론: - WHERE는 그룹핑 전 행 필터 / HAVING은 그룹핑 후 집계 필터 - SELECT 별칭은 WHERE/GROUP BY/HAVING에선 못 씀, ORDER BY에선 쓸 수 있음 - 집계함수(SUM,COUNT…)는 WHERE에 못 넣음 → HAVING - ⚠️ Oracle은 GROUP BY에 별칭 불가 → SELECT에서 만든 별칭 말고 식을 그대로 반복해야 함 (MySQL과 결정적 차이)

2. 기본 골격

SELECT   컬럼, 집계함수(...)
FROM     테이블 t                 -- Oracle 테이블 별칭엔 AS 안 씀 (컬럼 별칭엔 AS 가능)
JOIN     다른테이블 o ON t.id = o.id
WHERE    행_조건
GROUP BY 그룹_기준식
HAVING   집계_조건
ORDER BY 정렬_기준 [ASC|DESC];

3. 상위 N개 (⭐ Oracle의 최대 함정 — LIMIT 없음!)

-- ❌ 틀림: ROWNUM은 ORDER BY보다 먼저 매겨짐
SELECT * FROM PRODUCT WHERE ROWNUM <= 3 ORDER BY price DESC;

-- ✅ 방법 A (모든 버전): 정렬한 인라인 뷰를 ROWNUM으로 자르기
SELECT * FROM (
    SELECT name, price FROM PRODUCT ORDER BY price DESC
) WHERE ROWNUM <= 3;

-- ✅ 방법 B (12c+): FETCH FIRST
SELECT name, price FROM PRODUCT ORDER BY price DESC
FETCH FIRST 3 ROWS ONLY;

💡 "왜 ROWNUM<=3에 ORDER BY 붙이면 틀리나?"는 오라클 면접 단골. (ROWNUM은 정렬 전에 부여되기 때문)

4. JOIN

종류 의미
INNER JOIN 양쪽 다 있는 행만
LEFT JOIN 왼쪽 전부 + 매칭 안 되면 오른쪽 NULL 
-- 주문 없는 회원 (LEFT JOIN + IS NULL 국룰)
SELECT m.name
FROM MEMBER m
LEFT JOIN ORDERS o ON m.member_id = o.member_id
WHERE o.order_id IS NULL;

Oracle 구식 표기 (+)도 존재: WHERE m.member_id = o.member_id(+) (LEFT). 신문법(ANSI JOIN) 권장.

5. 집계함수

COUNT(*) 전체 / COUNT(col) NULL 제외 / COUNT(DISTINCT col) 중복 제외. SUM AVG MAX MIN.

SELECT category, COUNT(*) AS cnt, AVG(price) AS avg_price
FROM PRODUCT
GROUP BY category                 -- 실제 컬럼 OK
HAVING COUNT(*) >= 2
ORDER BY avg_price DESC;           -- ORDER BY엔 별칭 OK

6. 서브쿼리

-- 스칼라: 전체 평균보다 비싼 상품
SELECT name FROM PRODUCT
WHERE price > (SELECT AVG(price) FROM PRODUCT);

-- IN: 완료 주문한 회원
SELECT name FROM MEMBER
WHERE member_id IN (SELECT member_id FROM ORDERS WHERE status='완료');

-- 상관: 카테고리별 최고가 상품
SELECT p.* FROM PRODUCT p
WHERE p.price = (SELECT MAX(price) FROM PRODUCT x WHERE x.category = p.category);

7. 윈도우(분석)함수 — Oracle이 원조, 완전 지원

함수() OVER (PARTITION BY 그룹 ORDER BY 정렬)
함수 동점 처리
ROW_NUMBER() 1,2,3
RANK() 1,1,3
DENSE_RANK() 1,1,2 
-- 부서별 연봉 1위 (그룹별 TOP-N 정석)
SELECT * FROM (
  SELECT name, dept_id, salary,
         ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC) rn
  FROM EMPLOYEE
) WHERE rn = 1;

8. 날짜 (⭐ Oracle)

SYSDATE                              -- 현재 (MySQL의 NOW())
TO_CHAR(ordered_at, 'YYYY-MM-DD')    -- 포맷팅 (MySQL DATE_FORMAT)
TO_DATE('2024-06-01','YYYY-MM-DD')   -- 문자 → 날짜
DATE '2024-06-01'                    -- 날짜 리터럴 (권장)
EXTRACT(YEAR FROM d)                 -- 연도 (MySQL YEAR())
EXTRACT(MONTH FROM d)
d1 - d2                              -- 일수 차 (숫자)
d + 7                                -- 7일 뒤
MONTHS_BETWEEN(d1, d2)               -- 개월 차
-- 범위: WHERE d >= DATE '2024-06-01' AND d < DATE '2024-07-01'

⚠️ DATE 컬럼을 문자열('2024-06-01')과 직접 비교하면 NLS 설정 의존 → DATE 리터럴이나 TO_DATE로 명시.

9. 문자열 (⭐ Oracle)

a || '-' || b            -- 연결 (MySQL CONCAT), CONCAT은 인자 2개만
SUBSTR(s, 1, 3)          -- 1번째부터 3글자 (1-index)
LENGTH(s)                -- 글자 수 (Oracle은 문자 기준! 바이트는 LENGTHB)
INSTR(s, 'x')            -- 위치 찾기
UPPER LOWER TRIM
REPLACE(s, 'a', 'b')
LPAD(s, 5, '0')          -- 자리 채우기
s LIKE '김%'             -- 김으로 시작
REGEXP_LIKE(s, '^[0-9]+$')  -- 정규식 (MySQL의 REGEXP)

💡 MySQL과 달리 Oracle LENGTH는 이미 글자 수. (한글도 1글자=1)

10. NULL·조건 분기 (⭐ Oracle)

NVL(col, 0)                    -- NULL이면 0 (MySQL IFNULL)
NVL2(col, '있음', '없음')       -- NULL 아니면 첫째, NULL이면 둘째
COALESCE(a, b, c)              -- 처음 non-NULL (표준, Oracle도 OK)
CASE WHEN age < 20 THEN '10대'
     WHEN age < 30 THEN '20대'
     ELSE '30대+' END AS age_group
DECODE(gender,'M','남','F','여','기타')  -- Oracle 전용 간이 분기

⚠️ col = NULL (X) → col IS NULL (O).

11. 자주 하는 실수 TOP 8 (Oracle)

  1. LIMIT 습관 → Oracle엔 없음. ROWNUM 인라인뷰 / FETCH FIRST
  2. ROWNUM <= n 에 ORDER BY 직접 → 정렬 전에 잘림. 인라인뷰로 감싸기
  3. IFNULL → NVL, DATE_FORMAT → TO_CHAR, YEAR() → EXTRACT
  4. GROUP BY 별칭 사용 → Oracle 불가, 식을 반복
  5. 집계 조건을 WHERE에 → HAVING
  6. CONCAT(a,b,c) 3개 인자 → Oracle CONCAT은 2개만, || 사용
  7. DATE 컬럼을 문자열과 직접 비교 (NLS 의존)
  8. 정렬 동점 처리 안 함 → 2차 정렬 키 추가

문제 1. 서울 사는 회원 (나이 순)

MEMBER에서 서울에 사는 회원의 이름(name)과 나이(age)를, 나이가 어린 순으로 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT name, age
FROM MEMBER
WHERE city = '서울'
ORDER BY age ASC;

해설 - WHERE로 행을 먼저 거르고 ORDER BY로 정렬. ASC는 생략 가능(기본값)이지만 명시하면 실수 줄어듦. - 결과: 정하늘(19) → 김철수(25) → 박민수(41).

⚠️ 함정: 문자열 조건은 따옴표 필수('서울'). 나이 같은 사람이 있으면 순서가 흔들리므로, 실전에선 ORDER BY age, name처럼 2차 키를 붙이는 습관.

🔗 프로그래머스 대응: SELECT 카테고리 — "동물 보호소" 계열 기본 조회.

문제 2. 10만 원 초과 상품 (비싼 순)

PRODUCT에서 가격이 100000원을 초과하는 상품의 이름과 가격을, 비싼 순으로 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT name, price
FROM PRODUCT
WHERE price > 100000
ORDER BY price DESC;

해설 - 숫자 비교엔 따옴표 X. >(초과)와 >=(이상) 구분 — 지문의 "초과/이상/미만/이하"에 밑줄 긋는 습관. - 결과: 노트북(1,200,000) → 모니터(350,000).

⚠️ 함정: "초과"인데 >=로 쓰면 경계값 하나 틀림. LE는 점수를 안 보여주므로 이런 경계 실수를 스스로 잡아야 함.

🔗 프로그래머스 대응: SELECT — 조건 필터 + 정렬.

문제 3. 회원이 사는 도시 목록

MEMBER에 등장하는 도시(city)를 중복 없이 가나다순으로 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT DISTINCT city
FROM MEMBER
ORDER BY city ASC;

해설 - DISTINCT는 조회 결과 행 전체의 중복을 제거. 결과: 대구, 부산, 서울. - 여러 컬럼에 DISTINCT를 쓰면 "그 조합"이 유일한 행만 남음.

⚠️ 함정: DISTINCT는 특정 한 컬럼에만 거는 게 아니라 SELECT 목록 전체에 적용됨. SELECT DISTINCT a, b는 a만 유일이 아니라 (a,b) 조합이 유일.

🔗 프로그래머스 대응: SELECT — "중복 제거".

문제 4. 가장 비싼 상품 3개 (LIMIT)

PRODUCT에서 가장 비싼 상품 3개의 이름과 가격을 비싼 순으로 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle) — ⭐ Oracle엔 LIMIT이 없다

-- 방법 A (모든 버전): 정렬한 인라인 뷰를 ROWNUM으로 자르기
SELECT name, price
FROM (
    SELECT name, price FROM PRODUCT ORDER BY price DESC
)
WHERE ROWNUM <= 3;

-- 방법 B (Oracle 12c+): FETCH FIRST
SELECT name, price
FROM PRODUCT
ORDER BY price DESC
FETCH FIRST 3 ROWS ONLY;

해설 - 결과: 노트북, 모니터, 이어폰. - 핵심 함정: SELECT * FROM PRODUCT WHERE ROWNUM<=3 ORDER BY price DESC는 틀림. ROWNUM은 ORDER BY보다 먼저 부여되므로, 정렬 안 된 상태에서 3개를 자른 뒤 정렬함 → 엉뚱한 결과. - 그래서 정렬을 인라인 뷰 안에서 끝낸 뒤 바깥에서 ROWNUM으로 자르는 게 정석. (그룹별 상위 K는 윈도우함수 → 05)

⚠️ 함정: MySQL의 LIMIT 3 습관이 그대로 나가면 Oracle에서 문법 에러. "왜 ROWNUM에 ORDER BY 직접 쓰면 안 되나"는 오라클 면접 단골 질문이니 원리까지 기억.

SQL 02 — 집계 · GROUP BY · HAVING

경력직 SQL의 8할이 여기서 갈림. "행 필터(WHERE) vs 집계 필터(HAVING)"를 몸에 발라야 함.

문제 1. 카테고리별 상품 통계

PRODUCT에서 카테고리별 상품 수와 평균 가격을 구하되, 상품이 2개 이상인 카테고리만, 평균 가격이 비싼 순으로 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT category,
       COUNT(*)   AS cnt,
       AVG(price) AS avg_price
FROM PRODUCT
GROUP BY category
HAVING COUNT(*) >= 2
ORDER BY avg_price DESC;

해설 - GROUP BY category로 카테고리 단위로 묶고, 각 그룹에 집계함수 적용. - "2개 이상인 카테고리만"은 집계 결과에 대한 조건 → HAVING (WHERE 불가). - 결과: 전자(3개, 546,333) → 의류(2개, 39,000) → 도서(2개, 35,000). (식품은 1개라 제외)

⚠️ 함정: HAVING COUNT(*)>=2를 WHERE COUNT(*)>=2로 쓰면 에러. WHERE는 그룹핑 전이라 집계함수를 모름.

🔗 프로그래머스 대응: GROUP BY — "고양이와 개는 몇 마리", "카테고리별 도서 판매량".

문제 2. 회원별 완료 주문 건수

ORDERS에서 상태가 '완료'인 주문을 대상으로, 회원별 주문 건수를 많은 순으로 조회하세요. (건수 같으면 회원 ID 오름차순)

💡 정답 & 해설 보기

정답 (Oracle)

SELECT member_id, COUNT(*) AS order_cnt
FROM ORDERS
WHERE status = '완료'
GROUP BY member_id
ORDER BY order_cnt DESC, member_id ASC;

해설 - '완료'만 남기는 건 행 조건 → WHERE (그룹핑 전에 걸러야 빠르고 정확). - 그 다음 GROUP BY member_id로 세기. 결과: m1=2, m2=2, m3=1, m5=1. - 동점(2건)인 m1·m2는 2차 정렬 키 member_id로 순서 확정.

⚠️ 함정: WHERE와 HAVING을 헷갈리면 논리는 맞아도 성능/정확성이 흔들림. "행 조건=WHERE, 집계 조건=HAVING" 암기.

🔗 프로그래머스 대응: GROUP BY + 동점 정렬.

문제 3. 평균 연봉이 높은 부서

EMPLOYEE에서 부서(dept_id)가 있는 직원을 대상으로, 부서별 평균 연봉이 500만 원 이상인 부서를 평균 연봉 내림차순으로 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT dept_id, AVG(salary) AS avg_sal
FROM EMPLOYEE
WHERE dept_id IS NOT NULL
GROUP BY dept_id
HAVING AVG(salary) >= 5000000
ORDER BY avg_sal DESC;

해설 - WHERE dept_id IS NOT NULL = 사장(부서 없음) 제외 → 행 조건. - HAVING AVG(salary) >= 5000000 = 집계 결과 조건. - 결과: 개발(6,166,666) → 영업(5,900,000). 인사(4,500,000)는 미달로 제외.

⚠️ 함정: NULL은 = NULL이 아니라 IS NULL/IS NOT NULL. 그리고 이 필터는 집계 전이라 WHERE가 맞음.

🔗 프로그래머스 대응: GROUP BY + HAVING 조합(고득점 단골).

✅ 이 장 핵심

  • WHERE = 그룹핑 전 행 필터 / HAVING = 그룹핑 후 집계 필터
  • 집계함수: COUNT(*)(전체행), COUNT(col)(NULL 제외), SUM/AVG/MAX/MIN
  • 정렬은 항상 동점 2차 키까지

SQL 03 — JOIN

실무·시험 모두 최빈출. INNER vs LEFT, 그리고 LEFT JOIN + IS NULL(없는 것 찾기)이 핵심.

문제 1. 주문한 적 없는 회원 (LEFT JOIN + IS NULL)

한 번도 주문한 적이 없는 회원의 이름을 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT m.name
FROM MEMBER m
LEFT JOIN ORDERS o ON m.member_id = o.member_id
WHERE o.order_id IS NULL;

해설 - LEFT JOIN은 왼쪽(MEMBER) 전부를 남기고, 매칭되는 주문이 없으면 오른쪽을 NULL로 채움. - 매칭이 없던 회원만 골라내려면 WHERE o.order_id IS NULL. 결과: 강보라. - "~한 적 없는" = LEFT JOIN + IS NULL 은 통째로 암기하는 패턴.

⚠️ 함정: NOT IN (서브쿼리)로도 가능하지만, 서브쿼리에 NULL이 섞이면 결과가 통째로 사라지는 함정이 있음. LEFT JOIN 방식이 안전.

🔗 프로그래머스 대응: JOIN — "없어진 기록 찾기", "있었는데요 없었습니다".

문제 2. 완료 주문 내역 (INNER JOIN)

완료된 주문에 대해 주문번호, 주문한 회원 이름, 주문일을 주문번호 순으로 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT o.order_id, m.name, o.ordered_at
FROM ORDERS o
JOIN MEMBER m ON o.member_id = m.member_id
WHERE o.status = '완료'
ORDER BY o.order_id;

해설 - JOIN(=INNER JOIN)은 양쪽에 다 존재하는 행만. 여기선 모든 주문에 회원이 있으므로 결과 손실 없음. - 결과: 6건(주문 101,102,103,105,107,108). - 별칭(o, m)을 쓰면 컬럼 출처가 명확 → 다중 테이블에서 필수 습관.

⚠️ 함정: Oracle DATE 컬럼은 세션 설정에 따라 24/06/05 또는 시간까지 표시될 수 있음. 형식을 고정하려면 TO_CHAR(o.ordered_at, 'YYYY-MM-DD').

🔗 프로그래머스 대응: JOIN 기본.

문제 3. 상품별 총 판매 수량 (안 팔린 상품 포함)

모든 상품에 대해 총 판매 수량을 구하세요. 한 번도 안 팔린 상품은 0으로. 상품 ID 순으로 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT p.product_id, p.name,
       NVL(SUM(oi.quantity), 0) AS total_qty
FROM PRODUCT p
LEFT JOIN ORDER_ITEM oi ON p.product_id = oi.product_id
GROUP BY p.product_id, p.name
ORDER BY p.product_id;

해설 - "모든 상품"이 기준이므로 PRODUCT를 왼쪽에 두고 LEFT JOIN. - 안 팔린 상품은 SUM이 NULL → NVL(..., 0)으로 0 처리. (MySQL IFNULL의 오라클판) - 이 스키마에선 모든 상품이 팔렸지만, "0 포함" 요구는 LEFT JOIN + NVL 패턴을 쓰라는 신호.

⚠️ 함정: INNER JOIN으로 하면 안 팔린 상품이 결과에서 사라짐. "모든/전체 ~에 대해"라는 표현은 LEFT JOIN 신호.

🔗 프로그래머스 대응: JOIN + 집계.

문제 4. 회원별 총 결제 금액 (다중 조인)

완료된 주문 기준으로 회원별 총 결제 금액(수량 × 단가 합)을 많은 순으로 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT m.member_id, m.name,
       SUM(oi.quantity * p.price) AS total_amount
FROM MEMBER m
JOIN ORDERS o     ON m.member_id = o.member_id AND o.status = '완료'
JOIN ORDER_ITEM oi ON o.order_id = oi.order_id
JOIN PRODUCT p    ON oi.product_id = p.product_id
GROUP BY m.member_id, m.name
ORDER BY total_amount DESC;

해설 - 4개 테이블 체인 조인: 회원→주문→주문상세→상품. - 금액 = quantity * price의 합. 상태 필터를 JOIN의 ON 절에 넣어 완료 주문만. - 결과: 이영희(1,663,000) → 김철수(1,416,000) → 박민수(153,000) → 정하늘(25,000).

⚠️ 함정: 상태 조건을 WHERE o.status='완료'에 넣어도 이 경우엔 동일하지만, LEFT JOIN 상황에선 ON과 WHERE의 위치가 결과를 바꿈(WHERE에 두면 LEFT가 INNER처럼 동작). 금액 합은 큰 수라 실무에선 BIGINT/long 의식.

🔗 프로그래머스 대응: JOIN 다중 + 집계(고난도 단골).

✅ 이 장 핵심

  • INNER: 양쪽 다 있는 것 / LEFT: 왼쪽 전부 + 없으면 NULL
  • "~한 적 없는" → LEFT JOIN + IS NULL
  • "모든/전체 ~" → LEFT JOIN (+ NVL 0)
  • 다중 조인은 별칭 + ON 조건 위치 주의

SQL 04 — 서브쿼리

"평균보다 ~", "각 그룹의 최댓값 행" 같은 요구의 정답. 스칼라 / IN / 상관 서브쿼리 3종.

문제 1. 평균보다 비싼 상품 (스칼라 서브쿼리)

전체 상품 평균 가격보다 비싼 상품의 이름과 가격을 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT name, price
FROM PRODUCT
WHERE price > (SELECT AVG(price) FROM PRODUCT)
ORDER BY product_id;

해설 - 괄호 안 서브쿼리가 값 하나(스칼라)를 반환 → 그 값과 비교. - 전체 평균 226,625보다 비싼: 노트북, 모니터. - 평균/최대/최소 등 "전체 기준값과 비교"는 스칼라 서브쿼리.

⚠️ 함정: 스칼라 서브쿼리는 반드시 값 1개만 반환해야 함. 여러 행 반환하면 에러(= 대신 IN 필요).

🔗 프로그래머스 대응: "평균 이상/이하" 유형.

문제 2. 완료 주문을 한 회원 (IN 서브쿼리)

완료된 주문을 한 번이라도 한 회원의 이름을 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT name
FROM MEMBER
WHERE member_id IN (
    SELECT member_id FROM ORDERS WHERE status = '완료'
)
ORDER BY member_id;

해설 - 서브쿼리가 여러 값(집합)을 반환 → IN으로 포함 여부 판단. - 결과: 김철수, 이영희, 박민수, 정하늘. - JOIN + DISTINCT로도 가능하지만, "존재 여부"만 볼 땐 IN/EXISTS가 의도가 명확.

⚠️ 함정: NOT IN에 NULL이 섞이면 결과가 비어버림. "없는 것"은 NOT EXISTS나 LEFT JOIN이 안전(03).

🔗 프로그래머스 대응: 서브쿼리 / EXISTS.

문제 3. 카테고리별 최고가 상품 (상관 서브쿼리)

각 카테고리에서 가장 비싼 상품의 이름, 카테고리, 가격을 카테고리 순으로 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT p.name, p.category, p.price
FROM PRODUCT p
WHERE p.price = (
    SELECT MAX(x.price)
    FROM PRODUCT x
    WHERE x.category = p.category      -- 바깥 행을 참조 (상관)
)
ORDER BY p.category;

해설 - 서브쿼리가 바깥 쿼리의 각 행(p)을 참조 → 행마다 다시 평가되는 상관 서브쿼리. - "각 그룹의 최댓값을 가진 행"의 대표 패턴. 결과: 도서=자바의정석, 식품=커피원두, 의류=청바지, 전자=노트북.

⚠️ 함정: 최댓값이 동점이면 그 카테고리에서 여러 행이 나옴(의도일 수도, 아닐 수도). "정확히 1개"가 필요하면 윈도우함수 ROW_NUMBER(05)가 더 정확.

🔗 프로그래머스 대응: "카테고리별 최고가 상품"(고득점 키트 실제 문제와 유사).

✅ 이 장 핵심

  • 스칼라(값1): WHERE price > (SELECT AVG...)
  • IN(집합): 포함 여부
  • 상관 서브쿼리: 그룹별 최대/최소 행 — 바깥 행 참조가 핵심
  • "그룹별 정확히 1개"는 윈도우함수가 더 깔끔 → 다음 장

SQL 05 — 윈도우함수 (경력직 변별력)

"그룹별 순위 / 그룹별 TOP-N / 누적·이동 계산"의 정답. 신입은 몰라도 되지만 경력직은 알면 확 앞섬. 형식: 함수() OVER (PARTITION BY 그룹 ORDER BY 정렬)

문제 1. 부서별 연봉 순위

부서가 있는 직원에 대해 부서 안에서 연봉 순위를 매겨, 이름·부서·연봉·순위를 조회하세요. (부서, 순위 순)

💡 정답 & 해설 보기

정답 (Oracle)

SELECT name, dept_id, salary,
       RANK()       OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rnk,
       DENSE_RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS drnk
FROM EMPLOYEE
WHERE dept_id IS NOT NULL
ORDER BY dept_id, rnk;

해설 - PARTITION BY dept_id = 부서별로 창(window)을 나눔. ORDER BY salary DESC = 그 안에서 정렬. - 개발팀: 조개발(1) → 김개발(2) → 이코딩(3). - RANK vs DENSE_RANK vs ROW_NUMBER:

함수동점 처리 예 (100,100,90)
ROW_NUMBER()1,2,3 (동점도 강제 순번)
RANK()1,1,3 (다음 순위 건너뜀)
DENSE_RANK()1,1,2 (안 건너뜀)

⚠️ 함정: 순위 요구인데 LIMIT이나 서브쿼리로 우회하면 그룹별 처리가 지저분해짐. 윈도우함수가 정석.

문제 2. 부서별 연봉 1위 (그룹별 TOP-N)

각 부서에서 연봉이 가장 높은 직원 1명씩을 조회하세요. (부서 순)

💡 정답 & 해설 보기

정답 (Oracle)

SELECT name, dept_id, salary
FROM (
    SELECT name, dept_id, salary,
           ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rn
    FROM EMPLOYEE
    WHERE dept_id IS NOT NULL
) t
WHERE rn = 1
ORDER BY dept_id;

해설 - 윈도우함수는 WHERE에서 바로 못 씀(실행순서상 SELECT 이후). → 서브쿼리로 감싼 뒤 바깥에서 rn = 1. - 결과: 개발=조개발, 영업=윤영업, 인사=최인사. - 이 "감싸서 rn=1" 패턴이 그룹별 TOP-N의 표준 정답. TOP-3면 rn <= 3.

⚠️ 함정: 동점 1위가 둘인데 "정확히 1명"이 필요하면 ORDER BY salary DESC, emp_id처럼 tie-breaker를 넣어야 결정적.

문제 3. 부서 평균 대비 내 연봉 (집계 윈도우)

각 직원의 연봉이 자기 부서 평균 연봉과 얼마나 차이 나는지 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT name, dept_id, salary,
       AVG(salary) OVER (PARTITION BY dept_id)        AS dept_avg,
       salary - AVG(salary) OVER (PARTITION BY dept_id) AS diff
FROM EMPLOYEE
WHERE dept_id IS NOT NULL
ORDER BY dept_id, salary DESC;

해설 - 집계함수를 OVER와 함께 쓰면 행을 줄이지 않고 그룹 집계값을 각 행에 붙임(GROUP BY와 다름). - 개발팀 평균 6,166,666 → 조개발은 +1,833,334, 이코딩은 -1,166,666. - 이렇게 "개별 행 + 그룹 통계"를 한 번에 보여줘야 할 때 GROUP BY로는 불가, 윈도우가 유일.

⚠️ 함정: GROUP BY는 행을 그룹당 1개로 접지만, 윈도우 집계는 원본 행 수를 유지. 목적에 맞게 선택.

✅ 이 장 핵심

  • 함수() OVER (PARTITION BY 그룹 ORDER BY 정렬)
  • 순위 3종: ROW_NUMBER / RANK / DENSE_RANK (동점 처리 차이)
  • 그룹별 TOP-N = 서브쿼리로 감싸고 rn <= N
  • 집계 윈도우는 행을 안 접고 그룹 통계를 붙임

SQL 06 — 날짜 · 문자열 (⭐ Oracle)

MySQL과 함수 이름이 가장 많이 다른 파트. DATE_FORMAT·YEAR()·CONCAT은 오라클에서 안 통함. Oracle 기준으로 통일해서 외우세요.

문제 1. 특정 월의 주문 (날짜 범위)

2024년 6월에 접수된 주문의 주문번호를 주문번호 순으로 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

-- 방법 A: 범위 비교 (인덱스 유리, 권장) — DATE 리터럴 사용
SELECT order_id
FROM ORDERS
WHERE ordered_at >= DATE '2024-06-01'
  AND ordered_at <  DATE '2024-07-01'
ORDER BY order_id;

-- 방법 B: EXTRACT
SELECT order_id
FROM ORDERS
WHERE EXTRACT(YEAR FROM ordered_at) = 2024
  AND EXTRACT(MONTH FROM ordered_at) = 6
ORDER BY order_id;

해설 - 결과: 101,102,104,105,106,107 (6월 주문. 날짜 기준이므로 취소 주문 104도 포함). - 방법 A 권장: 컬럼에 함수를 안 씌워 인덱스를 탈 수 있음(성능 면접 포인트). - Oracle DATE엔 시간 성분이 있으므로, "6월 전체"는 >= 6/1 AND < 7/1이 안전(6/30 23:59도 포함).

⚠️ 함정 - MySQL의 YEAR(col)=2024는 오라클에서 EXTRACT(YEAR FROM col). 그리고 컬럼에 함수 씌우면 인덱스 무력화. - WHERE ordered_at >= '2024-06-01'처럼 문자열로 직접 비교하면 NLS_DATE_FORMAT에 의존해 에러/오작동 → DATE '...' 또는 TO_DATE로 명시.

🔗 프로그래머스 대응: Date 카테고리(Oracle 환경 선택).

문제 2. 가입 연도별 회원 수

가입 연도별 회원 수를 연도 오름차순으로 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT EXTRACT(YEAR FROM joined_at) AS join_year,
       COUNT(*) AS cnt
FROM MEMBER
GROUP BY EXTRACT(YEAR FROM joined_at)     -- 별칭 말고 식을 반복!
ORDER BY join_year;

해설 - EXTRACT(YEAR FROM d)로 연도 추출. 결과: 2021:1, 2022:1, 2023:2, 2024:2. - ⭐ Oracle은 GROUP BY join_year(별칭) 불가 → GROUP BY EXTRACT(YEAR FROM joined_at)로 식을 반복. (ORDER BY엔 별칭 OK) - 월별이면 TO_CHAR(joined_at, 'YYYY-MM')으로 묶기.

Oracle 날짜 함수 요약

SYSDATE                              -- 현재 (MySQL NOW())
EXTRACT(YEAR|MONTH|DAY FROM d)        -- 연/월/일 (MySQL YEAR/MONTH/DAY)
TO_CHAR(d, 'YYYY-MM-DD')             -- 포맷 (MySQL DATE_FORMAT)
TO_CHAR(d, 'YYYY-MM'), TO_CHAR(d,'DY')  -- 월/요일
TO_DATE('2024-06-01','YYYY-MM-DD')   -- 문자 → 날짜
d1 - d2                              -- 일수 차 (숫자)
ADD_MONTHS(d, 3), MONTHS_BETWEEN(d1,d2)
TRUNC(d)                             -- 시간 잘라 자정으로

⚠️ 함정: 출력 형식이 지문과 다르면 오답 처리 → TO_CHAR(col,'YYYY-MM-DD')로 감싸기. Oracle DATE는 시간 성분이 숨어 있으니 비교/집계 시 TRUNC 고려.

이름이 '김'으로 시작하는 회원의 이름과, 성을 뺀 부분(2번째 글자부터)을 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT name,
       SUBSTR(name, 2) AS without_surname
FROM MEMBER
WHERE name LIKE '김%'
ORDER BY member_id;

a || '-' || b            -- 연결 (CONCAT은 인자 2개만 → || 권장)

  • 날짜 범위: >= DATE '시작' AND < DATE '다음달1일' (인덱스 + 시간성분 안전)

NULL 처리와 조건 분기(CASE)는 잔실수 1순위. 마지막으로 그룹별 1위 뽑기를 종합.

문제 1. 나이대별 회원 수 (CASE)

회원을 10대 / 20대 / 30대 이상으로 나눠 각 구간의 회원 수를 조회하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT CASE
         WHEN age < 20 THEN '10대'
         WHEN age < 30 THEN '20대'
         ELSE '30대 이상'
       END AS age_group,
       COUNT(*) AS cnt
FROM MEMBER
GROUP BY CASE
           WHEN age < 20 THEN '10대'
           WHEN age < 30 THEN '20대'
           ELSE '30대 이상'
         END
ORDER BY age_group;

해설 - CASE WHEN은 위에서부터 처음 참인 조건을 채택 → 조건 순서가 중요(작은 값부터). - 결과: 10대 1명(정하늘), 20대 2명(김철수·최지은), 30대 이상 3명(이영희·박민수·강보라). - ⭐ Oracle은 GROUP BY에 별칭(age_group)을 못 씀 → CASE식을 그대로 반복해야 함. (단 ORDER BY엔 별칭 사용 가능) — MySQL과 결정적 차이.

⚠️ 함정: 구간 겹침 주의. WHEN age<30이 age<20보다 아래 있어야 10대가 20대에 안 먹힘.

🔗 프로그래머스 대응: "나이 정보가 없는 회원 수 구하기" 등 CASE/IS NULL 조합.

문제 2. 직원 부서명 표시 (LEFT JOIN + NULL 처리)

모든 직원의 이름과 소속 부서명을 조회하되, 부서가 없으면 '미배정'으로 표시하세요.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT e.name,
       COALESCE(d.dept_name, '미배정') AS dept_name
FROM EMPLOYEE e
LEFT JOIN DEPARTMENT d ON e.dept_id = d.dept_id
ORDER BY e.emp_id;

해설 - 직원 전부가 기준 → LEFT JOIN. 부서 없는 한사장은 d.dept_name이 NULL. - COALESCE(값, 대체값) = 처음 만나는 non-NULL 반환(다중 인자 가능). 2개면 Oracle NVL(값, 대체값)과 동일. - 결과: 한사장 → '미배정', 나머지는 소속 부서명.

⚠️ 함정: d.dept_name = NULL (X). NULL 비교는 IS NULL, 치환은 Oracle에선 NVL/COALESCE (IFNULL은 MySQL 전용이라 오라클에서 에러).

🔗 프로그래머스 대응: IS NULL 카테고리.

문제 3. 부서별 최고 연봉자 (그룹별 TOP-N 종합)

각 부서에서 연봉이 가장 높은 직원 1명을 뽑아 이름·부서·연봉을 조회하세요. 동점이면 사번(emp_id)이 빠른 사람.

💡 정답 & 해설 보기

정답 (Oracle)

SELECT name, dept_id, salary
FROM (
    SELECT name, dept_id, salary,
           ROW_NUMBER() OVER (
               PARTITION BY dept_id
               ORDER BY salary DESC, emp_id ASC   -- tie-breaker
           ) AS rn
    FROM EMPLOYEE
    WHERE dept_id IS NOT NULL
) t
WHERE rn = 1
ORDER BY dept_id;

해설 - 05의 그룹별 TOP-N 패턴 + 동점 tie-breaker(emp_id ASC)로 "정확히 1명" 보장. - 결과: 개발=조개발(8,000,000), 영업=윤영업(7,000,000), 인사=최인사(4,500,000).

⚠️ 함정: tie-breaker 없이 ORDER BY salary DESC만 쓰면 동점 시 누가 rn=1일지 비결정적 → 채점에서 흔들림.

🔗 프로그래머스 대응: JOIN/GROUP BY + 순위(고난도).

✅ 이 장 핵심

  • NULL: 비교는 IS NULL, 치환은 NVL/COALESCE
  • CASE WHEN은 조건 순서(작은 값부터), ELSE 잊지 말기
  • 그룹별 1위 = ROW_NUMBER + tie-breaker + rn=1

3. SQL 핵심 (경력직 최우선)

⭐ DBMS는 Oracle. 상세는 → sql/00_SQL_치트시트.md, sql/README.md

꼭 몸에 붙여야 할 7개 패턴 (Oracle): 1. JOIN — 없는 행까지 보려면 LEFT JOIN + IS NULL 2. GROUP BY … HAVING — 집계 후 조건은 HAVING. GROUP BY엔 별칭 불가(식 반복) 3. 상관 서브쿼리 — "각 그룹의 최댓값 행" 4. 윈도우함수 — ROW_NUMBER()/RANK() OVER(PARTITION BY … ORDER BY …) → 그룹별 TOP-N 5. 상위 N개 — LIMIT 없음! 정렬한 인라인뷰 + ROWNUM <= N, 또는 FETCH FIRST N ROWS ONLY(12c+) 6. 날짜/문자 — TO_CHAR, EXTRACT, TO_DATE, ||(연결), NVL, DECODE, REGEXP_LIKE 7. NULL → IS NULL / NVL(COALESCE) / CASE WHEN

⚠️ MySQL 습관(LIMIT,IFNULL,DATE_FORMAT,YEAR(),CONCAT 다인자) 그대로 쓰면 오라클에서 에러. ⏱ SQL은 "정답을 한 번에" 나와야 함 → 문법 암기 < 손 연습.

🎤 경력직 기술면접 · CS 노트

코테는 관문일 뿐, 경력직은 기술면접에서 결판. 9년차라면 "안다"를 넘어 "내 프로젝트에서 이렇게 썼다"로 답해야 함. 지하철에서 질문을 보고 머릿속으로 30초 답변을 굴려보는 용도.

1. 데이터베이스 (경력직 최다 출제)

인덱스

  • B-Tree 인덱스: 정렬된 트리, 범위 검색 O(log n). 대부분의 기본 인덱스.
  • 커버링 인덱스: 쿼리에 필요한 컬럼이 인덱스에 다 있어 테이블 접근(랜덤 I/O) 생략.
  • 인덱스가 안 타는 경우: 컬럼에 함수/연산(WHERE EXTRACT(YEAR FROM col)=2024), 앞부분 없는 LIKE '%x', 형변환, OR 남발, 낮은 카디널리티.
  • 💬 예상 Q: "인덱스를 걸었는데 왜 안 빨라졌나?" → 실행계획(EXPLAIN)에서 type=ALL(풀스캔) 확인, 함수/형변환 제거, 복합인덱스 순서.

트랜잭션 & 격리수준

  • ACID: 원자성/일관성/격리성/지속성.
  • 격리수준 4단계와 이상현상:
    수준Dirty ReadNon-repeatablePhantom
    READ UNCOMMITTEDOOO
    READ COMMITTEDXOO
    REPEATABLE READ (MySQL 기본)XXO*
    SERIALIZABLEXXX
  • *MySQL InnoDB는 갭락으로 팬텀도 상당 부분 방지.
  • 💬 예상 Q: "동시에 같은 재고를 차감하면?" → 트랜잭션 + 비관적 락(SELECT ... FOR UPDATE) 또는 낙관적 락(버전 컬럼).

정규화 vs 반정규화

  • 정규화: 중복 제거 → 무결성↑, 조인↑. 반정규화: 조회 성능 위해 의도적 중복.
  • 💬 "언제 반정규화?" → 읽기 압도적으로 많고 조인 비용이 병목일 때, 집계 컬럼 캐싱 등.

조인 알고리즘

  • Nested Loop(작은 집합), Hash Join(대량 등가조인), Sort-Merge(정렬된/범위).
  • N+1 문제: ORM에서 목록 조회 후 각 건마다 추가 쿼리 → fetch join/IN 절/배치로 해결.

2. 자료구조 · 알고리즘 (개념 확인용)

  • 시간복잡도 감각: 배열 접근 O(1), 탐색 O(n), 정렬 O(n log n), 해시 평균 O(1).
  • HashMap vs TreeMap: 해시(순서X, 평균 O(1)) vs 트리(정렬, O(log n)).
  • Stack/Queue/Deque/PriorityQueue 언제 쓰나 (→ 알고 치트시트).
  • 💬 "대용량 로그에서 상위 K개 빈도?" → HashMap 카운팅 + 크기 K 최소힙.

3. Java 심화 (주력 언어라면 필수)

  • GC: 객체 도달 불가 시 회수. Young(Minor GC)/Old(Major GC), G1GC 기본. Stop-the-world 개념.
  • HashMap 동작: 해시 버킷, 충돌 시 체이닝→트리화(8개↑), 리사이즈(부하율 0.75).
  • equals/hashCode 계약: equals 같으면 hashCode 같아야. 안 지키면 HashMap에서 유실.
  • == vs equals: 참조 vs 값. String 리터럴 풀.
  • 불변 객체(immutable): 스레드 안전, final 필드. String이 대표.
  • 동시성: synchronized, volatile(가시성), ConcurrentHashMap, AtomicInteger.
  • 💬 "ArrayList vs LinkedList" → 랜덤 접근 O(1) vs 중간 삽입/삭제 O(1)(단, 탐색 비용). 실무 대부분 ArrayList.

4. 설계 · 실무 (9년차 변별 구간)

  • REST 설계: 자원 중심 URI, HTTP 메서드 의미(GET 조회/POST 생성/PUT 전체수정/PATCH 부분/DELETE), 상태코드(2xx/4xx/5xx), 멱등성.
  • 트랜잭션 경계: 서비스 계층에 @Transactional, 외부 API 호출은 트랜잭션 밖으로, 전파 속성(REQUIRED/REQUIRES_NEW).
  • 캐시: 로컬(Caffeine) vs 분산(Redis), 캐시 무효화/스탬피드, TTL.
  • 대용량 처리: 페이징(오프셋 vs 커서/키셋), 벌크 연산, 배치.
  • 장애 대응 경험: 실제 겪은 성능 이슈 → 원인(쿼리/GC/락) → 해결 → 지표 개선을 수치로.
  • 💬 SI 특성상 "레거시 마이그레이션/대고객 시스템 운영/납기" 경험을 STAR(상황-과제-행동-결과)로 정리.

5. 경력 라이브 코딩 / 화이트보드 대비

  • 문제 받으면 바로 코딩 금지: (1) 입출력·제약 확인 (2) 예시로 이해 (3) 접근 말로 설명 (4) 복잡도 합의 (5) 코딩 (6) 엣지케이스 점검.
  • 생각을 소리 내어(thinking aloud): 면접관은 정답보다 사고 과정을 봄.
  • 막히면 "brute force부터 → 최적화" 순서로 진행.

6. 인성 / 컬처핏 (LE그룹)

  • 지원 동기: 왜 SI/LE인가(그룹 계열 도메인, 규모 있는 시스템, DX/클라우드 전환 등)를 본인 커리어와 연결.
  • 이직 사유: 부정적 표현 금지 → 성장/도전 프레이밍.
  • 협업·갈등 사례, 실패와 학습 사례 각 1개씩 STAR로 준비.
  • 💬 자소서 빈출: "지식·기술 역량을 지원 직무에서 어떻게 발휘할지"(300~700자) → 코테/이 노트 내용과 일관되게.

📦 부록 C — 연습 스키마 (Oracle)

모든 SQL 문제가 쓰는 연습용 테이블·데이터. Oracle Live SQL이나 dbfiddle(Oracle)에 통째로 붙여넣고 답지를 직접 돌려보세요.

-- ============================================================
--  LE SQL 연습용 스키마 (⭐ Oracle 기준)
--  전자상거래(회원/상품/주문) + 인사(부서/직원) 두 도메인.
--  모든 공략본 문제는 이 스키마를 사용합니다.
--  * 최초 실행 시 DROP에서 ORA-00942(테이블 없음)가 나면 무시하세요.
-- ============================================================

DROP TABLE ORDER_ITEM CASCADE CONSTRAINTS;
DROP TABLE ORDERS     CASCADE CONSTRAINTS;
DROP TABLE PRODUCT    CASCADE CONSTRAINTS;
DROP TABLE MEMBER     CASCADE CONSTRAINTS;
DROP TABLE EMPLOYEE   CASCADE CONSTRAINTS;
DROP TABLE DEPARTMENT CASCADE CONSTRAINTS;

-- 회원
CREATE TABLE MEMBER (
    member_id   NUMBER       PRIMARY KEY,
    name        VARCHAR2(30) NOT NULL,
    gender      CHAR(1),                 -- 'M' / 'F'
    age         NUMBER,
    city        VARCHAR2(20),
    joined_at   DATE
);

-- 상품
CREATE TABLE PRODUCT (
    product_id  NUMBER       PRIMARY KEY,
    name        VARCHAR2(50) NOT NULL,
    category    VARCHAR2(20),            -- 전자/도서/식품/의류
    price       NUMBER
);

-- 주문 (주문 헤더)
CREATE TABLE ORDERS (
    order_id    NUMBER PRIMARY KEY,
    member_id   NUMBER,
    ordered_at  DATE,
    status      VARCHAR2(10),            -- 완료/취소/배송중
    CONSTRAINT fk_orders_member FOREIGN KEY (member_id) REFERENCES MEMBER(member_id)
);

-- 주문 상세 (주문-상품 N:M)
CREATE TABLE ORDER_ITEM (
    order_id    NUMBER,
    product_id  NUMBER,
    quantity    NUMBER,
    CONSTRAINT pk_order_item PRIMARY KEY (order_id, product_id),
    CONSTRAINT fk_oi_order   FOREIGN KEY (order_id)   REFERENCES ORDERS(order_id),
    CONSTRAINT fk_oi_product FOREIGN KEY (product_id) REFERENCES PRODUCT(product_id)
);

-- 부서
CREATE TABLE DEPARTMENT (
    dept_id   NUMBER PRIMARY KEY,
    dept_name VARCHAR2(20)
);

-- 직원 (self-join: manager_id, NULL 포함)
CREATE TABLE EMPLOYEE (
    emp_id     NUMBER PRIMARY KEY,
    name       VARCHAR2(30),
    dept_id    NUMBER,          -- NULL 가능(사장)
    salary     NUMBER,
    manager_id NUMBER,          -- NULL 가능(사장)
    hired_at   DATE,
    CONSTRAINT fk_emp_dept FOREIGN KEY (dept_id) REFERENCES DEPARTMENT(dept_id)
);
